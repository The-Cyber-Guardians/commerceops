# پلن یادگیری بک‌اند — یک هفته‌ی TravelOps

> برای تیم ۳ نفره‌ی مبتدی. هدف: تا آخر هفته، تیم بتواند اسکلت بک‌اندِ TravelOps را با
> Django + DRF + PostgreSQL + JWT + Docker + تست بالا بیاورد و آماده‌ی «Iteration 0» شود.

---

## طرز استفاده از این پلن

* **پایه‌ی مشترک:** روزهای ۱ تا ۴ را **هر سه نفر** کامل کار می‌کنند (این‌ها ستون فقرات پروژه‌اند).
* **تخصص سبک:** از روز ۵ به بعد هر نفر روی حوزه‌ی خودش کمی عمیق‌تر می‌شود (طبق تقسیم‌کارِ `TravelOps-Guide.md`).
* **قانون طلایی:** هر مفهومی که یاد گرفتید، **همان روز روی خودِ پروژه** تمرین کنید — نه یک پروژه‌ی جدا. آخر هفته یک اسکلتِ واقعی دستتان است.
* **زمان فرضی:** ~۳ ساعت در روز برای هر نفر. اگر کمتر/بیشتر است، تعداد روزها را کش بدهید.
* **جفتی کار کنید (Pair):** حداقل نصف زمان را دو نفری روی یک صفحه کد بزنید؛ برای مبتدی‌ها سریع‌ترین راهِ یادگیری همین است.
* **منابع:** اول از همه **مستندات رسمی** (لینک‌ها پایین هر روز). ویدیو کمکی است، ولی مرجع اصلی مستندات رسمی باشد.

> **یادآوری از تصمیم‌ها (D5):** این هفته فقط بک‌اند + Swagger است. تنها بخشی که باید «درست و جدی» یاد گرفته شود
> **JWT + Permissions** است؛ بقیه در حدِ کارراه‌انداز کافی‌ست.

## روز ۱ — Python لازم، محیط کار، و Git

**هدف روز:** همه بتوانند یک محیط تمیز بسازند و با Git راحت کار کنند.

**یاد بگیرید:**
* مرور Python: توابع، کلاس‌ها، لیست/دیکشنری، list comprehension، ماژول و import، virtual environment (`venv`)، `pip` و `requirements.txt`.
* Git در حدِ کار تیمی: `clone`, `branch`, `add`, `commit`, `push`, `pull`, و مفهومِ Pull Request و Code Review.

**تمرین روی پروژه:**
1. همه ریپو را `clone` کنند، `venv` بسازند و فعال کنند.
2. یک برنچ آزمایشی بزنید، فایل کوچکی اضافه کنید، PR بدهید و یکی از اعضا Review کند و merge کند.
3. `requirements.txt` اولیه با `django` و `djangorestframework` بسازید.

**منابع:** مستند رسمی Python (بخش venv)، Pro Git book (فصل‌های ۱–۳)، مستند GitHub درباره‌ی Pull Request.

**چک‌لیست پایان روز:** هر ۳ نفر یک PR موفق زده و merge کرده‌اند ✅

---

## روز ۲ — Django: مدل‌ها، ORM و Migration

**هدف روز:** فهمِ قلب Django — چطور دیتابیس با کدِ پایتون توصیف می‌شود.

**یاد بگیرید:**
* ساختار پروژه‌ی Django (`startproject`, `startapp`)، `settings.py`، `urls.py`.
* Model و فیلدها، `ForeignKey`/`OneToOne`/`ManyToMany`.
* `makemigrations` و `migrate` (اول با SQLite پیش‌فرض، Postgres روز ۴).
* Django Admin برای دیدن داده‌ها.
* Django ORM: `.objects.create/filter/get/all`.

**تمرین روی پروژه:**
1. اپ‌های `accounts` و `tours` را بسازید.
2. مدل‌های ساده طبق سند: `Organization`, `Tour` (با فیلد `status` = DRAFT/PUBLISHED/CLOSED — طبق D4)، `Destination`.
3. Migration بگیرید و از طریق Admin چند تور نمونه بسازید.

**منابع:** آموزش رسمی Django (بخش‌های Models و Admin از tutorial رسمی).

**چک‌لیست:** توانستید یک Tour در حالت DRAFT بسازید و در Admin ببینید ✅

---

## روز ۳ — Django REST Framework: ساخت API

**هدف روز:** تبدیل مدل‌ها به API واقعی (JSON).

**یاد بگیرید:**
* Serializer (مثل مترجمِ بین مدل و JSON).
* `APIView` و مهم‌تر: `ViewSet` + `Router`.
* عملیات CRUD روی یک endpoint.
* مفهوم Permission در DRF (فعلاً سطح ساده؛ عمیقش روز ۴).

**تمرین روی پروژه:**
1. برای `Tour` یک `TourSerializer` و `TourViewSet` بسازید.
2. endpointهای `GET /tours/` و `POST /tours/` را بالا بیاورید.
3. قانونِ D4 را پیاده کنید: در لیستِ عمومی فقط تورهای `PUBLISHED` برگردند (تورهای DRAFT پنهان بمانند).

**منابع:** مستند رسمی DRF (بخش Tutorial: Serializers, ViewSets & Routers).

**چک‌لیست:** با یک درخواست HTTP توانستید تور بسازید و فقط تورهای Published را در لیست ببینید ✅

## روز ۴ — PostgreSQL + JWT (مهم‌ترین روز)

**هدف روز:** دیتابیس واقعی + احراز هویت درست. (این روز را جدی بگیرید — طبق D5 قلبِ پروژه همین است.)

**یاد بگیرید:**
* PostgreSQL: نصب/اجرا، ساختِ دیتابیس و کاربر، اتصال Django به Postgres در `settings.py`.
* JWT: مفهوم Access/Refresh Token و اینکه چرا stateless است.
* کتابخانه‌ی `djangorestframework-simplejwt`: endpointهای `token/` و `token/refresh/`.
* Permissionها: `IsAuthenticated` و ساختِ یک Permission سفارشی ساده بر اساس نقش (آژانس/مشتری — طبق D2).

**تمرین روی پروژه:**
1. دیتابیس را از SQLite به Postgres سوییچ کنید و دوباره `migrate` بگیرید.
2. لاگین با JWT را فعال کنید؛ توکن بگیرید و با آن به endpointهای Tour دسترسی بزنید.
3. یک قانون ساده: ساختِ تور فقط برای کاربرِ «آژانس» مجاز باشد، نه «مشتری».

**منابع:** مستند رسمی Django (بخش Databases/PostgreSQL)، مستند `djangorestframework-simplejwt`.

**چک‌لیست:** بدون توکن ۴۰۱ می‌گیرید؛ با توکنِ آژانس می‌توانید تور بسازید ✅

## روز ۵ — Docker و Docker Compose

**هدف روز:** کل پروژه با یک دستور بالا بیاید تا هر سه نفر محیط یکسان داشته باشند.

**یاد بگیرید:**
* مفهوم Image و Container.
* `Dockerfile` برای اپ Django.
* `docker-compose.yml` برای بالا آوردن همزمانِ **web (Django) + db (Postgres)**.
* متغیرهای محیطی (`.env`) برای رمز دیتابیس و تنظیمات.

**تمرین روی پروژه:**
1. `Dockerfile` و `docker-compose.yml` بنویسید (سرویس web + سرویس db).
2. کاری کنید `docker compose up` پروژه را بالا بیاورد و migrateها اجرا شوند.
3. تست کنید که از داخل کانتینر web به db وصل می‌شوید.

**منابع:** مستند رسمی Docker (Get Started) و راهنمای رسمیِ «Django in Docker Compose».

**چک‌لیست:** روی سیستمِ هر سه نفر `docker compose up` پروژه را بالا می‌آورد ✅

**(تخصص عضو A):** این روز مالِ توست — کمی عمیق‌تر روی Docker و بعداً CI.

## روز ۶ — تست (Pytest) + Ruff + Swagger

**هدف روز:** کیفیت و مستندسازی خودکار.

**یاد بگیرید:**
* `pytest` + `pytest-django`: نوشتنِ تست ساده، فیکسچر، تستِ API با `APIClient`.
* `Ruff`: لینت و فرمت؛ اجرا و رفعِ خطاها.
* `drf-spectacular` (یا معادل): تولید خودکارِ Swagger/OpenAPI و صفحه‌ی مستند تعاملی.

**تمرین روی پروژه:**
1. دو تست بنویسید: (الف) تورِ DRAFT در لیست عمومی دیده نشود؛ (ب) بدون توکن ساختِ تور ۴۰۱ بدهد.
2. `ruff check` را تمیز کنید.
3. صفحه‌ی Swagger را بالا بیاورید و endpointها را از داخلش صدا بزنید.

**منابع:** مستند رسمی Pytest و pytest-django، مستند Ruff، مستند drf-spectacular.

**چک‌لیست:** `pytest` سبز، `ruff` بدون خطا، صفحه‌ی `/api/schema/swagger-ui/` باز می‌شود ✅

**(تخصص عضو C):** روی تستِ سناریوهای رزرو/ظرفیت (D3) بیشتر تمرکز کن — همزمانی و پرداخت.

---

## روز ۷ — GitHub Actions (CI) + جمع‌بندی

**هدف روز:** خودکارسازیِ تست روی هر PR و یک دموی کامل.

**یاد بگیرید:**
* مفهوم CI و فایل workflow در `.github/workflows/`.
* یک workflow که روی هر Push/PR: نصب وابستگی‌ها → `ruff check` → `pytest` را اجرا کند.

**تمرین روی پروژه:**
1. یک workflow ساده بسازید که تست و لینت را اجرا کند.
2. یک PR الکی بزنید و ببینید CI سبز/قرمز می‌شود.
3. **دموی end-to-end** (همگی با هم): از طریق Swagger این مسیر کوتاه را طی کنید —
   ساختِ کاربر آژانس → لاگین (JWT) → ساختِ Tour در DRAFT → Publish → دیدنش در لیست عمومی.

**منابع:** مستند رسمی GitHub Actions (Quickstart) + نمونه‌ی workflow برای Django/Postgres.

**چک‌لیست:** CI روی PR سبز می‌شود و دموی end-to-end بدون خطا اجرا می‌شود ✅

**(تخصص عضو A):** CI مالِ توست.

## بعد از این هفته چه می‌شود؟

اگر چک‌لیست‌ها سبز باشند، تیم آماده‌ی **Iteration 0** است: شروعِ پیاده‌سازیِ واقعیِ ماژول‌ها طبق `TravelOps-Guide.md`.
مبحث‌های سنگین‌تر (Multi-Tenancy با Base Manager، و قفلِ همزمانیِ ظرفیت روی مرحله‌ی پرداخت — طبق D3) عمداً برای
حینِ کدنویسیِ واقعی و به‌صورت جفتی گذاشته شده‌اند، نه این هفته‌ی پایه.

## نکته‌ها برای مبتدی‌ها
* **گیر کردید؟** قانون ۱۵ دقیقه: ۱۵ دقیقه خودت تلاش کن، بعد از تیم بپرس. وقتِ کسی را با گیرِ چندساعته تلف نکنید.
* **کوچک commit کنید:** هر تکه‌ی کوچکِ کارکننده را جدا commit کنید؛ برای مرور و برگشت راحت‌تر است.
* **همه‌چیز را یاد نگیرید:** هدفِ این هفته «راه افتادن» است، نه استادی. عمق بعداً حین پروژه می‌آید.
