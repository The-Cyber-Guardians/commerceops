# CommerceOps — نسخه ساده‌شده و قابل‌اجرا

> برای یک تیم ۳ نفره مبتدی. بدون ددلاین. هدف: کارآموزی.

## ۱. چه چیزی را نگه می‌داریم و چه چیزی را حذف می‌کنیم

### نگه‌داشتن (تأثیرگذار + قابل‌یادگیری)

| ویژگی | چرا نگه داریم |
|---|---|
| JWT Auth (ورود/رفرش/خروج) | همه شرکت‌ها از این استفاده می‌کنند. طلای مصاحبه. |
| PostgreSQL + Migrations | مهارت واقعی دیتابیس، نه پروژه‌های اسباب‌بازی با SQLite |
| Docker Compose | هر آگهی استخدامی Docker می‌خواهد |
| REST API با DRF | استاندارد صنعت |
| Git + GitHub Flow | جریان کاری حرفه‌ای |
| چندمستأجری پایه (Multi-tenancy) | نشان می‌دهد جداسازی داده را می‌فهمید |
| مدل ساده سفارش | منطق اصلی کسب‌وکار |
| Pytest + CI پایه | تست برای رزومه ضروری است |
| مستندات Swagger/OpenAPI | API شما را حرفه‌ای نشان می‌دهد |

### حذف (خیلی پیچیده برای مبتدی‌ها)

| ویژگی | چرا حذف می‌شود | به‌جایش چه کنیم |
|---|---|---|
| قفل همزمانی رزرو (Concurrent reservation locking) | `select_for_update` + جلوگیری از deadlock سخت است | بررسی ساده: اگر موجودی >= تعداد، رزرو کن. وجود race condition را بپذیر. |
| Idempotency (جدول IdempotencyRecord) | ماشین حالت پیچیده برای کلیدها | فقط مستندش کن. پیاده‌سازی را رد کن. |
| Webhook پرداخت + شبیه‌ساز | پیچیدگی یکپارچه‌سازی خارجی | یک toggle بولی ساده: علامت‌گذاری سفارش به‌عنوان پرداخت‌شده با یک فراخوانی مستقیم API |
| تسک‌های async با Celery | تنظیم broker، تلاش مجدد، مدیریت وضعیت تسک | همه‌چیز را سینکرون انجام بده. بدون تسک پس‌زمینه. |
| RBAC چند-نقشی (۶ نقش) | ماتریس دسترسی بزرگ است | فقط ۲ نقش: مدیر (Admin) و عضو (Member) |
| Audit Log (فقط‌افزایشی) | جدول اضافه، پیچیدگی اضافه | کلاً حذف کن |
| ۱۲ ماژول | یادگیری و ساخت خیلی زیاد | ادغام در ۵ ماژول ساده‌تر |
| امنیت سطح ردیف (Row-level security) | PostgreSQL پیشرفته | فقط با فیلتر org_id در کوئری‌ها |
| چرخه کامل تحویل سفارش (Fulfillment) | Picking → Packing → Shipment زیاد است | ساده: علامت‌گذاری سفارش به‌عنوان «ارسال‌شده» با یک شماره رهگیری |
| لاگ‌گیری ساختاریافته JSON | خوب است ولی ضروری نیست | لاگ‌گیری استاندارد Django |
| mypy + بررسی نوع (type checking) | منحنی یادگیری تند برای مبتدی‌ها | فعلاً رد کن، بعداً اگر وقت بود اضافه کن |
| مستندات ADR | برای یک پروژه کارآموزی زیاده‌روی است | فقط یک README که تصمیمات را توضیح دهد |

## ۳. چیزی که واقعاً می‌سازیم (ساده‌شده)

یک API بک‌اند که یک کسب‌وکار کوچک بتواند:
1. **ثبت‌نام / ورود** با JWT
2. **ایجاد محصولات** با تنوع‌ها (variants) و قیمت‌ها
3. **مدیریت موجودی** (افزودن/کاهش انبار)
4. **ایجاد سفارش‌ها** (انتخاب مشتری، محصولات، تعداد)
5. **علامت‌گذاری سفارش‌ها** به‌عنوان پرداخت‌شده / ارسال‌شده / تحویل‌داده‌شده
6. **مشاهده گزارش‌ها** (تعداد سفارش‌ها، هشدارهای کمبود موجودی)

**همین.** تمیز، ساده، قابل‌ارائه.

## ۴. ساختار ساده‌شده ماژول‌ها

```
commerceops/
├── src/
│   ├── manage.py
│   ├── config/          # تنظیمات Django، آدرس‌ها
│   ├── accounts/        # کاربر، احراز هویت JWT، نقش‌های ساده
│   ├── products/        # محصول، تنوع، انبار، موجودی
│   ├── orders/          # مشتری، سفارش، آیتم سفارش، وضعیت پرداخت
│   ├── reports/         # اندپوینت‌های کوئری ساده
│   └── common/          # ابزارهای مشترک، مدل‌های پایه
├── tests/
├── docker/
├── compose.yaml
├── pyproject.toml
└── README.md
```

**۵ ماژول به‌جای ۱۲ ماژول.** هر نفر مالک تقریباً ۲ ماژول است.

## ۵. مدل داده ساده‌شده

```
User (id, email, password, role, org_id)
    │
Organization (id, name, slug)
    │
Product (id, org_id, name, description, is_active)
    │
ProductVariant (id, product_id, org_id, sku, name, price, stock_quantity)
    │
Customer (id, org_id, name, email, phone)
    │
Order (id, org_id, customer_id, status, total, created_at)
    │
OrderItem (id, order_id, variant_id, quantity, unit_price)
```

**ساده‌سازی‌های کلیدی:**
- موجودی فقط یک عدد صحیح روی `ProductVariant` است (بدون جدول جداگانه InventoryItem)
- بدون مدل جداگانه Reservation (موجودی هنگام ایجاد سفارش کسر می‌شود)
- بدون مدل‌های Fulfillment/Shipment (فقط یک فیلد وضعیت سفارش)
- بدون مدل Address (فعلاً رد کن)
- بدون ردگیری StockMovement

**وضعیت‌های سفارش:** `PENDING` → `PAID` → `SHIPPED` → `DELIVERED` (یا `CANCELLED`)

## ۶. کدهای وضعیت HTTP

| کد | زمان استفاده |
|---|---|
| 200 | موفقیت (GET, PATCH) |
| 201 | ایجادشده (POST) |
| 400 | درخواست نامعتبر / خطای اعتبارسنجی |
| 401 | وارد نشده |
| 402 | وارد شده اما بدون دسترسی |
| 404 | یافت نشد |
| 409 | تعارض (مثلاً موجودی ناکافی) |

## ۷. نحوه تقسیم کار (۳ نفر)

### عضو A — احراز هویت و زیرساخت (تمرکز هفته ۱ تا ۲)
```
ماژول accounts/:
- مدل User سفارشی (ورود بر اساس ایمیل)
- راه‌اندازی JWT (ثبت‌نام، ورود، رفرش، خروج)
- دسترسی ساده بر اساس نقش (admin در برابر member)
- مدل Organization (نام، slug)
- راه‌اندازی Docker Compose
- پایپ‌لاین CI
```

### عضو B — محصولات و موجودی (تمرکز هفته ۲ تا ۳)
```
ماژول products/:
- CRUD محصول
- CRUD تنوع محصول (ProductVariant)
- انبار (Warehouse) (مدل ساده: نام، موقعیت)
- مدیریت موجودی (دریافت موجودی، بررسی در دسترس‌بودن)
- اندپوینت هشدار کمبود موجودی
```

### عضو C — سفارش‌ها و گزارش‌ها (تمرکز هفته ۳ تا ۴)
```
ماژول orders/:
- مدل Customer
- ایجاد سفارش (با اعتبارسنجی موجودی)
- تغییر وضعیت سفارش (پرداخت، ارسال، تحویل، لغو)
- لیست سفارش‌ها با فیلتر
- اندپوینت‌های گزارش (خلاصه سفارش‌ها، لیست کمبود موجودی)
```

### بعد از هفته ۴ — همگی با هم
- تست یکپارچگی (جریان کامل: ایجاد کاربر → افزودن محصول → ایجاد سفارش → پرداخت → ارسال)
- README با دستورالعمل راه‌اندازی، مستندات API، اسکرین‌شات‌ها
- مستندات Swagger/OpenAPI
- پرداخت نهایی و داده‌های نمایشی

## ۸. برنامه یادگیری (گام‌به‌گام)

- Serializers

- ورود/خروج/رفرش

### گام ۴: PostgreSQL (۱ روز)
- نصب و اتصال
- فهم migrations
- کوئری‌های پایه

### گام ۵: Docker (۱ تا ۲ روز)
- `Dockerfile`

- اتصال سرویس‌ها

### گام ۶: ساخت پروژه
- ترتیب ماژول‌های بالا را دنبال کنید

## ۹. چه چیزی این پروژه را برای کارآموزی متمایز می‌کند

حتی در حالت ساده‌شده، این پروژه نشان می‌دهد:

| مهارت | چگونه |
|---|---|
| **احراز هویت JWT** | جریان کامل ورود/رفرش/خروج |
| **طراحی REST API** | الگوهای URL یکسان، کدهای وضعیت درست، مدیریت خطا |
| **PostgreSQL** | دیتابیس واقعی (نه SQLite)، migrations، محدودیت‌ها |
| **Docker** | اپلیکیشن چند-سرویسی کانتینری‌شده |
| **جریان کاری Git** | برنچ‌ها، PR‌ها، کامیت‌های معنادار |
| **تست** | مجموعه تست Pytest با تست‌های API |
| **CI/CD** | پایپ‌لاین GitHub Actions |
| **معماری تمیز** | ماژول‌های جدا با مسئولیت‌های مشخص |
| **منطق کسب‌وکار** | جریان سفارش، اعتبارسنجی موجودی، تغییر وضعیت |
| **مدیریت خطا** | پاسخ‌های خطای ساختاریافته، نه کرش کردن |

### README که تأثیرگذار است
شامل کنید:
1. کارکرد پروژه (یک پاراگراف)
2. استک فناوری با بج‌ها (badges)
3. نحوه اجرا (دستورات کپی-پیست)
4. مستندات API (جدول اندپوینت‌ها)
5. دیاگرام معماری (حتی ساده)
6. چیزی که یاد گرفتید

## ۱۰. چیزهایی که بعداً اگر وقت داشتید اضافه کنید

فقط بعد از اینکه هسته اصلی کار کرد:

1. **کش Redis** — کش لیست محصولات
2. **Celery** — اعلان‌های ایمیل به‌صورت async
3. **نقش‌های بیشتر** — اپراتور انبار، مالی
4. **Audit log** — پیگیری اینکه چه کسی چه کاری کرده
5. **محافظت همزمانی** — `select_for_update` روی موجودی
6. **Idempotency** — جلوگیری از سفارش‌های تکراری
7. **mypy** — بررسی نوع استاتیک

**این‌ها را ابتدا اضافه نکنید.** چیز ساده را بسازید، کارش را درست کنید، تمیزش کنید، بعد پیچیدگی اضافه کنید.
