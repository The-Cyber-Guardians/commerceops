# CommerceOps — نسخه ساده‌شده و قابل‌اجرا

> مناسب برای یک تیم سه‌نفره مبتدی، بدون زمان‌بندی سخت و با هدف آماده‌شدن برای کارآموزی.

---

## ۱. چه چیزهایی را نگه می‌داریم و چه چیزهایی را حذف می‌کنیم؟

### مواردی که نگه می‌داریم

این موارد هم ارزش آموزشی دارند و هم در یک پروژه رزومه‌ای قابل ارائه‌اند.

| قابلیت | دلیل نگه‌داشتن |
|---|---|
| احراز هویت مبتنی بر JWT؛ شامل ورود، نوسازی توکن و خروج | یکی از روش‌های رایج احراز هویت در رابط‌های برنامه‌نویسی و موضوعی مناسب برای مصاحبه فنی است. |
| PostgreSQL و مهاجرت‌های پایگاه داده | مهارت کار با یک پایگاه داده واقعی را نشان می‌دهد و پروژه را از نمونه‌های ساده مبتنی بر SQLite متمایز می‌کند. |
| Docker Compose | اجرای یکسان پروژه روی دستگاه اعضای تیم را ساده می‌کند. |
| رابط برنامه‌نویسی REST با Django REST Framework | ساختار اصلی ارتباط میان فرانت‌اند و بک‌اند پروژه است. |
| Git و جریان کاری GitHub | همکاری تیمی، شاخه‌سازی، بازبینی کد و ادغام تغییرات را تمرین می‌کنید. |
| چندسازمانی پایه | نشان می‌دهد جداسازی داده‌های کسب‌وکارهای مختلف را درک می‌کنید. |
| مدل ساده سفارش | هسته منطق کسب‌وکار پروژه را تشکیل می‌دهد. |
| Pytest و یکپارچه‌سازی پیوسته پایه | وجود تست خودکار و اجرای آن در فرایند ادغام، پروژه را حرفه‌ای‌تر می‌کند. |
| مستندات Swagger/OpenAPI | استفاده و بررسی رابط برنامه‌نویسی را برای اعضای تیم و ارزیاب ساده‌تر می‌کند. |

### مواردی که فعلاً حذف می‌کنیم

| قابلیت | دلیل حذف | جایگزین ساده |
|---|---|---|
| قفل‌گذاری هم‌زمان برای رزرو موجودی | استفاده درست از `select_for_update`، مدیریت تراکنش و جلوگیری از بن‌بست برای شروع پروژه سنگین است. | بررسی کنید موجودی از تعداد درخواستی کمتر نباشد و سپس سفارش را ثبت کنید. وجود احتمال رقابت هم‌زمان را در محدودیت‌های پروژه توضیح دهید. |
| تکرارناپذیری درخواست‌ها یا `Idempotency` | نیازمند طراحی کلید، وضعیت درخواست و پاسخ ذخیره‌شده است. | مفهوم را در بخش محدودیت‌ها توضیح دهید و فعلاً پیاده‌سازی نکنید. |
| وب‌هوک پرداخت و شبیه‌ساز درگاه | یکپارچه‌سازی خارجی، اعتبارسنجی امضا و مدیریت تلاش مجدد Scope پروژه را بزرگ می‌کند. | یک عملیات مستقیم برای علامت‌گذاری سفارش به‌عنوان پرداخت‌شده بسازید. |
| وظایف غیرهم‌زمان با Celery | نیازمند کارگزار پیام، Worker، تلاش مجدد و مدیریت شکست است. | همه عملیات نسخه اول را هم‌زمان اجرا کنید. |
| کنترل دسترسی با نقش‌های متعدد | ماتریس دسترسی بزرگ و تست آن دشوار می‌شود. | فقط دو نقش `Admin` و `Member` داشته باشید. |
| گزارش تغییرات یا Audit Log | جدول و قواعد نگهداری تاریخچه بیشتری نیاز دارد. | در نسخه اول حذف شود. |
| دوازده ماژول جداگانه | یادگیری، هماهنگی و نگهداری پروژه را سنگین می‌کند. | قابلیت‌ها را در پنج ماژول اصلی ادغام کنید. |
| امنیت در سطح ردیف پایگاه داده | از قابلیت‌های پیشرفته PostgreSQL است و طراحی نادرست آن خطرناک است. | همه پرس‌وجوهای تجاری را با `org_id` محدود کنید. |
| چرخه کامل آماده‌سازی و ارسال سفارش | مراحل برداشت، بسته‌بندی، ارسال و رهگیری برای نسخه اول زیاد است. | سفارش فقط با یک شماره رهگیری به وضعیت ارسال‌شده منتقل شود. |
| ثبت رویدادها با قالب JSON | مفید است، اما برای نسخه اولیه ضروری نیست. | از سامانه ثبت رویداد استاندارد Django استفاده کنید. |
| بررسی ایستای نوع‌ها با mypy | برای تیم تازه‌کار منحنی یادگیری جداگانه‌ای دارد. | فعلاً حذف شود و در صورت باقی‌ماندن زمان بررسی شود. |
| سندهای مستقل تصمیم معماری یا ADR | برای این اندازه پروژه هزینه نگهداری زیادی دارد. | تصمیم‌های مهم در یک بخش مشخص از `README.md` ثبت شوند. |

## ۳. محصولی که واقعاً می‌سازیم

یک رابط برنامه‌نویسی بک‌اند می‌سازیم که یک کسب‌وکار کوچک بتواند:

1. با JWT ثبت‌نام و وارد سامانه شود.
2. محصول، گونه محصول و قیمت تعریف کند.
3. موجودی کالا را افزایش یا کاهش دهد.
4. برای یک مشتری سفارش ایجاد کند.
5. وضعیت سفارش را به پرداخت‌شده، ارسال‌شده یا تحویل‌داده‌شده تغییر دهد.
6. گزارش خلاصه سفارش‌ها و هشدار کمبود موجودی را مشاهده کند.

هدف نسخه اول همین مسیر کامل است: کوچک، تمیز، قابل اجرا و قابل ارائه.

## ۵. مدل ساده داده

```text
User
(id, email, password, role, org_id)
    │
Organization
(id, name, slug)
    │
Product
(id, org_id, name, description, is_active)
    │
ProductVariant
(id, product_id, org_id, sku, name, price, stock_quantity)
    │
Customer
(id, org_id, name, email, phone)
    │
Order
(id, org_id, customer_id, status, total, created_at)
    │
OrderItem
(id, order_id, variant_id, quantity, unit_price)
```

### ساده‌سازی‌های اصلی

- موجودی فقط یک عدد صحیح در `ProductVariant` است و جدول جداگانه‌ای برای موجودی نداریم.
- مدل جداگانه‌ای برای رزرو موجودی نداریم و موجودی هنگام ایجاد سفارش کاهش پیدا می‌کند.
- مدل‌های جداگانه آماده‌سازی، بسته‌بندی و ارسال نداریم و وضعیت سفارش برای نمایش این مراحل کافی است.
- در نسخه اول مدل نشانی یا Address نداریم.
- تغییرات موجودی در جدول جداگانه‌ای ثبت نمی‌شوند.

### وضعیت‌های سفارش

```text
PENDING → PAID → SHIPPED → DELIVERED
```

مسیر لغو:

```text
PENDING → CANCELLED
```

تیم باید پیش از پیاده‌سازی مشخص کند که لغو سفارش در چه وضعیت‌هایی مجاز است و آیا موجودی در زمان لغو بازگردانده می‌شود یا خیر.

## ۶. کدهای وضعیت HTTP

| کد | زمان استفاده |
|---|---|
| `200 OK` | درخواست موفق برای خواندن یا ویرایش |
| `201 Created` | منبع جدید با موفقیت ایجاد شده است |
| `400 Bad Request` | داده ورودی نامعتبر یا خطای اعتبارسنجی |
| `401 Unauthorized` | کاربر احراز هویت نشده یا توکن معتبر ندارد |
| `403 Forbidden` | کاربر احراز هویت شده، اما اجازه انجام عملیات را ندارد |
| `404 Not Found` | منبع پیدا نشده یا کاربر مجاز به مشاهده آن نیست |
| `409 Conflict` | درخواست با وضعیت فعلی منبع تعارض دارد؛ مانند موجودی ناکافی |

> نکته: کد `402 Payment Required` برای وضعیت «کاربر وارد شده ولی دسترسی ندارد» مناسب نیست. برای این حالت از `403 Forbidden` استفاده کنید.

## ۷. تقسیم کار تیم سه‌نفره

این تقسیم‌بندی برای شروع است و به معنای مالکیت دائمی بخش‌ها نیست. هر تغییر باید دست‌کم توسط یک عضو دیگر بازبینی شود.

### عضو A — احراز هویت و زیرساخت

تمرکز پیشنهادی در مرحله اول:

```text
accounts/
├── مدل User سفارشی با ورود مبتنی بر ایمیل
├── ثبت‌نام
├── ورود
├── نوسازی توکن
├── خروج
├── نقش‌های Admin و Member
└── Organization
```

همچنین:

- راه‌اندازی Docker Compose
- متغیرهای محیطی
- اتصال PostgreSQL
- خط پایه GitHub Actions

### عضو B — محصولات و موجودی

```text
products/
├── ایجاد، مشاهده، ویرایش و حذف محصول
├── ایجاد، مشاهده، ویرایش و حذف ProductVariant
├── قیمت و SKU
├── افزایش و کاهش موجودی
├── بررسی موجودی در دسترس
└── هشدار کمبود موجودی
```

اگر مدل Warehouse در پروژه استفاده می‌شود، باید ارتباط آن با `stock_quantity` روشن باشد. وجود هم‌زمان Warehouse و یک عدد موجودی مستقیم، بدون تعریف رابطه مشخص، باعث تناقض در مدل داده می‌شود.

### عضو C — سفارش و گزارش

```text
orders/
├── مدل Customer
├── ساخت سفارش
├── اعتبارسنجی موجودی
├── محاسبه مبلغ کل
├── تغییر وضعیت سفارش
├── فیلتر و فهرست سفارش‌ها
└── لغو سفارش
```

```text
reports/
├── خلاصه تعداد و مبلغ سفارش‌ها
└── فهرست محصولات کم‌موجودی
```

### پس از تکمیل بخش‌های اصلی

هر سه عضو روی این موارد همکاری می‌کنند:

- تست یکپارچه مسیر کامل
- هماهنگی Frontend و Backend
- مستندات Swagger/OpenAPI
- داده‌های نمایشی
- بازبینی مجوزها و جداسازی سازمان‌ها
- آماده‌سازی `README.md`
- اصلاح تجربه نمایش پروژه

مسیر اصلی آزمایش یکپارچه:

```text
ساخت سازمان و کاربر
→ ساخت محصول
→ افزودن موجودی
→ ثبت مشتری
→ ایجاد سفارش
→ پرداخت
→ ارسال
→ تحویل
```

## ۸. مسیر یادگیری گام‌به‌گام

### گام اول: مبانی Django

موضوعات:

- مدل‌ها
- مهاجرت‌ها
- پنل مدیریت
- تنظیمات
- مسیرها
- دستورهای `python manage.py`
- Django Shell

خروجی مورد انتظار:

- یک مدل ساده بسازید.
- Migration ایجاد و اجرا کنید.
- داده را از Admin و Shell مدیریت کنید.

### گام دوم: مبانی Django REST Framework

موضوعات:

- Serializer
- ViewSet
- Router
- Permission
- پاسخ و کد وضعیت
- عملیات `GET`، `POST`، `PATCH` و `DELETE`

خروجی مورد انتظار:

- یک منبع ساده را با API ایجاد، خواندن و ویرایش کنید.
- خطاهای ورودی را به‌شکل کنترل‌شده برگردانید.

### گام سوم: احراز هویت JWT

موضوعات:

- Access Token
- Refresh Token
- تنظیم Simple JWT
- ورود و نوسازی توکن
- خروج و بی‌اعتبارکردن Refresh Token
- Permission Class

خروجی مورد انتظار:

- کاربر وارد شود.
- به مسیر محافظت‌شده دسترسی پیدا کند.
- Access Token را نوسازی کند.
- خروج او رفتار مشخص و تست‌شده‌ای داشته باشد.

### گام چهارم: PostgreSQL

موضوعات:

- اتصال Django به PostgreSQL
- Migration
- محدودیت‌های `unique` و `check`
- Foreign Key
- تراکنش پایه
- Query و Index مقدماتی

### گام پنجم: Docker

موضوعات:

- `Dockerfile`
- سرویس‌های `app` و `db`
- Volume
- متغیرهای محیطی
- اجرای `docker compose up`
- اجرای Migration داخل Container

### گام ششم: ساخت پروژه

- ماژول‌ها را به ترتیب برنامه اجرا کنید.
- هر قابلیت را در یک شاخه جدا توسعه دهید.
- هم‌زمان با هر قابلیت تست بنویسید.
- Pull Request کوچک بسازید.
- تغییرها را مرتب روی GitHub قرار دهید.
- ادغام بخش‌ها را به پایان پروژه موکول نکنید.

## ۱۰. قابلیت‌هایی که بعداً می‌توان اضافه کرد

این قابلیت‌ها فقط بعد از کامل، تست‌شده و قابل ارائه‌شدن هسته اصلی بررسی شوند.

1. **Redis** برای ذخیره موقت فهرست محصولات
2. **Celery** برای ارسال اعلان یا ایمیل به‌شکل غیرهم‌زمان
3. نقش‌های بیشتر مانند مسئول موجودی یا مسئول مالی
4. ثبت تاریخچه تغییرات یا Audit Log
5. حفاظت در برابر رقابت هم‌زمان با `select_for_update`
6. تکرارناپذیری درخواست‌ها برای جلوگیری از سفارش تکراری
7. بررسی ایستای نوع‌ها با mypy
8. ثبت تغییرات موجودی با مدل `StockMovement`
9. استقرار نسخه نمایشی
10. بررسی کارایی Queryها و مشکل `N+1`

قانون پروژه:

> ابتدا نسخه ساده را بسازید، عملکرد آن را با تست ثابت کنید، کد را تمیز کنید و فقط سپس پیچیدگی جدید اضافه کنید.
